# 음악 추천 미니앱 PRD

**서비스명(가칭):** 오늘의 한 곡  
**플랫폼:** Toss 앱 내 App in Toss 미니앱  
**문서 버전:** v1.0  
**작성 목적:** MVP 출시 및 향후 유저 참여형 음악 추천 서비스 확장 고려

---

## 1. 서비스 요약

토스 앱 내에서 가볍게 소비할 수 있는 서비스는 진입 장벽이 낮고, 반복 방문을 만들기 좋다.  
본 서비스는 “오늘 추천 음악 보기”라는 단순한 행동을 중심으로, 사용자가 하루 1회 자연스럽게 방문하도록 만드는 **초경량 음악 큐레이션 미니앱**이다.

핵심 경험은 다음과 같다.

- 유저는 하루에 1곡~최대 3곡의 추천곡을 확인한다.
- 추천곡 확인 직전에 광고를 노출하여 수익을 만든다.
- 추천곡에는 앨범 표지와 간단한 메타데이터가 함께 제공된다.
- 유저는 원하는 스트리밍 플랫폼으로 즉시 이동해 곡을 재생할 수 있다.
- 장기적으로는 “사연과 함께 신청곡 제출”, “유료 신청곡 반영”, “실시간 곡 교체” 같은 참여형 경험으로 확장한다.

---

## 2. 배경

음악 추천 서비스는 많지만, 대부분 진입 후 선택지가 너무 많거나 큐레이션의 무게가 무겁다.  
반대로 유저는 종종 아래 니즈를 가진다.

- “오늘 뭐 듣지?”를 빠르게 해결하고 싶다.
- 남이 엄선한 한두 곡 정도는 부담 없이 받아들이고 싶다.
- 발견한 좋은 곡을 바로 내가 쓰는 플랫폼에서 듣고 싶다.
- 나중에는 내가 좋아하는 곡을 다른 사람에게 소개하고 싶다.

즉, 이 서비스는 **탐색 피로를 줄이고 즉시 소비 가능한 단일 추천 경험**을 제공하는 것이 핵심이다.

## 3. 목표

### 3.1 비즈니스 목표
- 광고 기반 수익 창출
- 앱인토스 내 반복 방문 유도
- 향후 유저 참여형/유료 기능으로 확장 가능한 기반 확보

### 3.2 제품 목표
- 하루 1회 이상 방문해도 부담 없는 가벼운 UX 제공
- 추천곡 확인까지의 플로우를 매우 짧게 유지
- 추천곡에서 실제 스트리밍 재생까지의 전환율 극대화
- 초기에는 운영자가 큐레이션한 곡 중심으로 안정적인 품질 확보

### 3.3 성공 지표
- DAU / WAU
- 추천 보기 버튼 CTR
- 광고 노출 완료율
- 추천곡 노출 후 플랫폼 이동 CTR
- 플랫폼별 이동 비중
- 날짜별 재방문율 (D1, D7, D30)
- 추천곡 노출당 스트리밍 전환율
- 추후 신청곡 제출률 / 유료 전환율

## 4. 타겟 유저

### 핵심 타겟

- 특정 노래를 다른 사람에게 들려주고 싶은 유저
- 유료로라도 자신이 원하는 곡을 전면에 띄우고 싶은 유저

## 5. 사용자 시나리오

### 시나리오 A. 오늘의 추천곡 보기
1. 유저가 미니앱 진입
2. 메인 화면에서 “오늘의 추천음악 보기” 버튼 클릭
3. 전면 광고 노출
4. 광고 종료 후 오늘 날짜에 해당하는 추천곡 1~3곡 표시
5. 유저가 표지/곡명/아티스트 확인
6. 원하는 스트리밍 플랫폼 버튼 클릭
7. 외부 앱 또는 웹 링크로 이동
8. 해당 플랫폼에서 곡 재생

### 시나리오 B. 추천곡이 여러 개일 때
1. 유저가 광고 시청 후 결과 화면 진입
2. 오늘의 추천곡 3개가 카드형으로 노출
3. 유저는 각 곡마다 다른 플랫폼 버튼을 누를 수 있음

### 시나리오 C. 확장 기능: 신청곡 제출
1. 유저가 “나도 곡 추천하기” 진입
2. 사연 + 곡 정보 입력
3. 제출
4. 운영 검수 후 편성 또는 반려

### 시나리오 D. 확장 기능: 유료 곡 교체
1. 유저가 “지금 이 순간 내 곡 띄우기” 진입
2. 곡 정보 입력
3. 결제
4. 결제 성공 시 현재 노출곡 교체 또는 예약 반영
5. 반영 결과 확인

## 9. 기능 명세

### 9.1 추천곡 조회
- 시스템은 현재 날짜 기준으로 playlist DB를 조회한다.
- 해당 날짜에 매핑된 곡 1~3개를 반환한다.
- 곡이 없으면 빈 상태 화면을 노출한다.

### 9.2 광고 노출
- 유저가 CTA 클릭 시 광고 노출을 시도한다.
- 광고 종료 이벤트 이후 결과를 노출한다.
- 광고 실패 시 fallback 정책 수행한다.

### 9.3 플랫폼 이동
- 각 곡 카드에는 4개 플랫폼 버튼이 존재한다.
- 버튼 클릭 시 해당 링크를 외부 URL 열기 기능으로 실행한다.
- 링크가 앱 스킴 또는 외부 웹 URL일 수 있다.

### 9.4 운영 관리
- 운영자는 날짜별 플레이리스트를 사전 등록한다.
- 각 곡 레코드에는 앨범 이미지 URL과 플랫폼별 링크를 포함한다.
- 같은 날짜에 최대 3곡까지 저장 가능하다.

## 11. 추천곡 / 플랫폼 링크 처리 전략

### 옵션 A. 플랫폼 링크를 DB에 직접 저장
#### 장점
- 초기 구현이 단순하다.
- 런타임 API 비용이 없다.
- 날짜 기반 정적 큐레이션과 잘 맞는다.
- 플랫폼별 정책 차이를 운영으로 흡수할 수 있다.

#### 단점
- 운영자가 직접 링크를 관리해야 한다.
- 곡 수가 늘어나면 수동 관리 비용이 늘어난다.

### 옵션 B. 플랫폼 API를 호출해 동적으로 생성
#### 장점
- 운영 입력이 줄어들 수 있다.
- 일부 플랫폼에서 메타데이터 갱신 자동화 가능하다.

#### 단점
- 플랫폼별 API 정책이 다르다.
- 일부 플랫폼은 공개 API 활용이 어렵다.
- 비용, rate limit, 인증, 유지보수 복잡도가 증가한다.

### 최종 권장안
**MVP는 플랫폼 링크와 이미지 URL을 모두 DB에 저장하는 방식을 권장한다.**

이유는 다음과 같다.
- 현재 서비스는 개인화 실시간 검색이 아니라 날짜별 고정 추천 구조다.
- 곡 수가 제한적이어서 수동 또는 반자동 운영이 가능하다.
- 플랫폼별 정책과 제약이 달라 API 일원화 효율이 낮다.
- 초기에는 비용보다 운영 복잡도 절감이 더 중요하다.

## 14. 운영 정책

### 14.1 곡 편성 정책
- 하루 최소 1곡, 최대 3곡
- 최소 2주~1개월치 사전 등록 권장
- 이미지 URL과 각 플랫폼 링크 검수 후 publish
- 링크 누락 시 해당 플랫폼 버튼 비활성화 가능

### 14.2 빈 상태 정책
- 오늘 날짜 편성곡이 없으면 “오늘의 추천곡을 준비 중이에요” 노출
- 전일 추천곡 재노출은 초기에는 비권장

### 14.3 링크 실패 정책
- 특정 플랫폼 링크 실패 시 토스트 노출
- 가능하면 앱 스킴 + 웹 fallback 저장 권장

### 14.4 광고 실패 정책
- 광고 실패 시 결과는 그대로 노출
- 수익보다 초기 리텐션 확보를 우선한다

## 15. 리스크 및 대응

### 리스크 1. 광고 이탈
- 광고를 보기 싫어 바로 이탈할 수 있다.
- 대응: CTA 문구로 기대값을 안내하고, 결과 화면 만족도를 높인다.

### 리스크 2. 외부 플랫폼 링크 실패
- 사용자 기기 환경에 따라 앱 설치/로그인 여부가 다르다.
- 대응: 앱 스킴 + 웹 fallback 전략, 운영 사전 검수, 실패 토스트 제공

### 리스크 3. 운영 비용 증가
- 날짜별로 곡/이미지/링크를 계속 입력해야 한다.
- 대응: 초기 수동 운영 후, 내부 CMS 또는 CSV 업로드 도입 검토

### 리스크 4. 유료 곡 교체 악용
- 부적절한 곡이 즉시 노출될 수 있다.
- 대응: 검수 단계, 금칙어 정책, 예약 반영 모델 우선 적용

## 18. 최종 권장안

이 서비스의 MVP는 **“광고 보고, 오늘의 추천곡 1~3개를 보고, 원하는 플랫폼에서 즉시 듣는 경험”** 하나에 집중하는 것이 적합하다.

초기 구현은 다음 방향을 권장한다.

- App in Toss + Supabase 기반으로 구현
- 광고는 버튼 클릭 직후 노출
- 날짜별 편성 데이터는 운영자가 사전 등록
- 앨범 이미지와 플랫폼 링크는 모두 DB에 저장
- 플랫폼 이동은 외부 URL 또는 딥링크 방식으로 처리
- 향후 신청곡, 유료 교체, 로그인/결제 기능을 확장할 수 있도록 테이블 구조를 분리 설계

이 구조는 MVP를 빠르게 출시하면서도 이후 참여형 서비스로 자연스럽게 확장하기에 적합하다.
